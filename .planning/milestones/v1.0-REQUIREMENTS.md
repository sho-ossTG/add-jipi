# Requirements — Server A → D Integration

**Version:** v1.0
**Date:** 2026-02-28
**Source:** PROJECT.md + research synthesis

---

## Goal

Adapt Server A to route all episode resolution through Server D (central data middleware) instead of calling Server B directly. A becomes thin: receive request → call D → return result with episode title. Three side channels are added (UA forwarding, failure log shipping). D does not exist yet — A must stub gracefully until D is live.

---

## Functional Requirements

### FR-1: D Client Module
**Must have.** Create `modules/integrations/d-client.js` that:
- Exposes `createDClient(options)` factory matching broker-client's injectable pattern
- Implements `resolveEpisode(episodeId)` → `{ url, title }` via `POST /api/resolve` to `D_BASE_URL`
- Throws structured errors (`code: "dependency_unavailable"`) when D_BASE_URL is unset or D is unreachable
- Uses `executeBoundedDependency` with tight timeouts: 5s attempt / 10s total (not the broker's 60s)
- Validates response shape: `url` must be HTTPS string, `title` must be non-empty string; throws `validation_error` on failure

**Acceptance:** Requesting when D_BASE_URL is unset → stream response is degraded (not a crash or 500). Requesting with D returning `{ url, title }` → stream payload contains the title.

---

### FR-2: Wire D Client as Resolution Transport
**Must have.** Replace broker-client in the resolution path:
- Update `resolveEpisodeResolver` in `modules/routing/stream-route.js` to use `createDClient`
- Update `addon.js` to import and instantiate `createDClient` instead of `createBrokerClient`
- Remove `cleanTitle` / `resolveBrokerFilename` title extraction helpers (dead code after migration)
- Use `response.title` verbatim in `formatStream(title, url)` — no local transformation

**Acceptance:** Stremio receives a stream payload with the episode title sourced from D's response (not parsed from URL filename). Existing degradation behavior is preserved when D fails.

---

### FR-3: Broker Client Deprecation
**Must have.** `modules/integrations/broker-client.js` call sites removed. File kept on disk until D is confirmed live in production (Phase 5).

**Acceptance:** No code imports or calls `createBrokerClient` after Phase 2. `broker-client.js` file still exists (reference implementation) but has zero callers.

---

### FR-4: User-Agent Forwarding
**Must have.** Forward raw User-Agent header to D after each successful stream resolution:
- Fire-and-forget: not awaited in critical path, never propagates errors
- Payload: `{ userAgent, episodeId, timestamp }` → `POST /api/ua` to `D_BASE_URL`
- Observable failure: emit `warn`-level log on failure (not silent swallow); optionally increment `ua_forward_error` reliability counter
- No-op when `D_BASE_URL` is unset

**Acceptance:** UA is forwarded to D on each resolution. A stream failure due to D being unavailable does NOT affect UA forwarding (and vice versa).

---

### FR-5: Failure Log Shipping (Nightly)
**Must have.** During the nightly shutdown window, ship `quarantine:events` from Redis to D:
- Triggered alongside (but after) existing nightly rollup — rollup runs first, then log shipping
- Non-destructive read: use `LRANGE quarantine:events 0 -1` (not LPOP) — rollup owns cleanup
- Payload: `{ source: "server-a", shippedAt, events: [...] }` → `POST /api/logs` to `D_BASE_URL`
- Fire-and-forget at the rollup level: failure logged at WARN, never blocks rollup completion
- No-op when `D_BASE_URL` is unset

**Acceptance:** After a nightly trigger, if D is reachable, A's quarantine events appear in D's log. If D is unreachable, the nightly rollup still completes without error.

---

### FR-6: D Client HTTP Contract (Documentation)
**Must have.** Before Phase 2 wiring, the D HTTP contract must be defined and documented:
- `POST /api/resolve` — request/response shape, error codes, timeout expectations
- `POST /api/ua` — request shape, acceptable responses
- `POST /api/logs` — request shape, acceptable responses

This contract is A's implementation spec and D's interface requirement. It must exist before any resolution wiring is merged.

**Acceptance:** Contract document exists and is referenced by both A's D client and D's implementation work.

## Non-Functional Requirements

### NFR-1: No New Dependencies
All integration work uses patterns already in the codebase (native fetch, `executeBoundedDependency`, env-var stub mode). No new npm packages.

### NFR-2: Zero Regression on Existing Policy Gates
Time window policy, session capacity gate, and operator authentication must behave identically before and after this integration. These code paths must not be modified.

### NFR-3: Graceful Degradation When D Is Unreachable
A must return a valid (degraded) Stremio response when D is unavailable — same behavior as when the broker was unavailable. No 500s, no uncaught exceptions.

### NFR-4: Observable Failure Channels
Side channels (UA forwarding, log shipping) must produce `warn`-level log entries on failure. Silent empty catches are not acceptable.

### NFR-5: D Timeout Budget
D client total timeout must not exceed 10 seconds. This prevents D unavailability from holding Redis session slots and exhausting function execution budget.

## Prerequisites (must complete before integration code)

### PRE-1: Wire Unwired Tests
These four test files are not wired to any npm script and never run in CI:
- `tests/analytics-hourly.test.js`
- `tests/analytics-nightly-rollup.test.js`
- `tests/session-view-ttl.test.js`
- `tests/request-controls-nightly.test.js`

These cover the exact paths this integration modifies. Wire them before writing any integration code.

### PRE-2: Fix Nightly Rollup Bugs
Three known bugs must be fixed before log shipping tests are written:
- Bug 1: Analytics gap in shutdown window
- Bug 2: HyperLogLog unimplemented (unique count)
- Bug 3: Daily rollup unique count always zero

Writing log shipping tests against broken rollup behavior encodes bugs as expected behavior.

### PRE-3: Extract Shared `executeBoundedDependency`
Currently duplicated in `broker-client.js` and `redis-client.js`. Extract to `modules/integrations/bounded-dependency.js` before adding a third copy in `d-client.js`.

## Out of Scope

- Offloading hourly analytics or session tracking to D — stays in A's Redis layer
- Building Server B, C, or D
- Changes to policy gates, session management, or operator authentication routes
- Stremio catalog handler changes
- Non-Stremio clients or mobile apps
- Vercel Cron configuration for automated nightly trigger (manual operator route is sufficient for v1)

---

## Success Criteria

- [ ] Stream requests resolve with episode title sourced from D's response (not filename parsing)
- [ ] When D is unreachable, A returns a valid degraded stream — no 500s
- [ ] User-Agent is forwarded to D on each successful resolution
- [ ] Nightly rollup ships failure logs to D without blocking rollup completion
- [ ] No broker-client callers remain in the codebase
- [ ] All four previously-unwired tests run in CI
- [ ] Existing session gate, time window, and operator route behavior unchanged
