# Roadmap — Server A → D Integration

**Project:** Adapt Server A to route all episode resolution through Server D instead of calling Server B (broker) directly.
**Goal:** A becomes thin — receive request, call D, return stream with episode title. Three side channels added: UA forwarding, failure log shipping, and broker deprecation.

## Phase 1: D Client Interface + Stub

**Goal:** `modules/integrations/d-client.js` exists, all three functions are implemented with correct stub and error behavior, the D HTTP contract is documented, and nothing in the live request path has changed. Safe to deploy at any point in this phase.

**Deliverables:**

- Document the D HTTP contract: `POST /api/resolve`, `POST /api/ua`, `POST /api/logs` — request/response shapes, error codes, timeout expectations (FR-6)
- Create `modules/integrations/d-client.js` exposing `createDClient(options)` factory (FR-1)
  - `resolveEpisode(episodeId)` → `{ url, title }` via `POST /api/resolve`; throws `dependency_unavailable` when `D_BASE_URL` is unset; throws `validation_error` when response fails shape check (`url` must be HTTPS string, `title` must be non-empty string)
  - `forwardUserAgent(userAgent, episodeId)` → fire-and-forget `POST /api/ua`; silent no-op when `D_BASE_URL` is unset
  - `shipFailureLogs(events)` → fire-and-forget `POST /api/logs`; silent no-op when `D_BASE_URL` is unset
  - Uses `executeBoundedDependency` from `modules/integrations/bounded-dependency.js` with 5s attempt / 10s total timeouts (not broker's 60s)
- Create `tests/contract-d-client.test.js` covering (FR-1):
  - Stub behavior: `resolveEpisode` throws structured error when `D_BASE_URL` unset
  - `forwardUserAgent` and `shipFailureLogs` are silent no-ops when `D_BASE_URL` unset
  - Error classification: timeout → `dependency_timeout`; 503 → `dependency_unavailable`; malformed response → `validation_error`
  - Response validation: missing `url` field throws; non-HTTPS `url` throws; missing `title` field throws; empty `title` throws

**Success criteria:**

- `D_BASE_URL` unset + stream request → response is degraded (valid Stremio payload), not a 500 or crash
- `D_BASE_URL` set + D returns `{ url, title }` → stream payload contains the title field from D
- All three error classes (`dependency_timeout`, `dependency_unavailable`, `validation_error`) produce thrown errors with correct `code` property, verified by tests
- `forwardUserAgent` called when `D_BASE_URL` is unset → no error thrown, no network call made
- `modules/routing/stream-route.js`, `addon.js`, and all other request-path files are unchanged from Phase 0

**Dependencies:** Phase 0 complete (shared `executeBoundedDependency` module must exist before `d-client.js` imports it).

## Phase 3: User-Agent Forwarding

**Goal:** Each successful stream resolution fires a UA forward to D as a background side channel. The call is never awaited in the critical path and its failure never affects the stream response.

**Deliverables:**

- Add `dClient.forwardUserAgent(userAgent, episodeId)` call in the success branch of `modules/routing/stream-route.js`, after `resolveEpisode` returns and before `sendJson` (FR-4)
- Call is not awaited — wrap in an immediately-invoked promise that catches all errors
- On failure: emit a `warn`-level log entry (not an empty catch); optionally increment `ua_forward_error` reliability counter (FR-4)
- Add or update tests in `tests/contract-d-client.test.js` (or a dedicated stream integration test) to assert: (FR-4)
  - UA forward is not awaited (stream response returns before forward resolves)
  - UA forward error does not propagate to stream response
  - `warn`-level log is emitted on UA forward failure

**Success criteria:**

- A stream resolution completes and the response is returned to the Stremio client before the UA forward POST resolves
- Simulating a UA forward timeout or 500 from D does not change the stream response status or body
- A `warn`-level log entry appears when the UA forward call fails — no silent empty catch
- `D_BASE_URL` unset → no network call made, no error thrown, stream response unaffected

**Dependencies:** Phase 2 complete (`stream-route.js` must already use `dClient` before UA forwarding is added to it; `dClient.forwardUserAgent` is already defined in Phase 1).

**Plans:** 2/2 plans complete

## Phase 4: Nightly Log Shipping

**Goal:** During the nightly shutdown window, A ships accumulated `quarantine:events` from Redis to D after the rollup completes. Rollup completion is never blocked by log shipping success or failure.

**Deliverables:**

- Add `dClient.shipFailureLogs(events)` trigger in `modules/routing/operator-routes.js` nightly handler (FR-5)
- Sequencing: rollup runs to completion first; log shipping is called after, not concurrently (FR-5)
- Redis read is non-destructive: use `LRANGE quarantine:events 0 -1`; rollup owns cleanup of the list (FR-5)
- Failure handling: `shipFailureLogs` errors are caught at the rollup handler level, logged at `warn`, and do not cause the nightly route to return an error response (FR-5)
- Optionally add `failuresShipped` count to the nightly rollup response body (FR-5)
- Add tests asserting: (FR-5)
  - Log shipping runs after rollup (not before, not concurrently)
  - `shipFailureLogs` failure does not change the nightly route response status
  - `LRANGE` is used (not `LPOP`) — `quarantine:events` list is unchanged after shipping

**Success criteria:**

- After a nightly trigger with D reachable: D's `/api/logs` receives a payload containing A's quarantine events with `source: "server-a"`
- After a nightly trigger with D unreachable: the nightly rollup route returns its normal success response; a `warn`-level log entry is present for the shipping failure
- `quarantine:events` Redis list is not modified by the shipping operation (rollup retains ownership of cleanup)
- Log shipping never starts before rollup finishes in any test or production scenario

**Dependencies:** Phase 3 complete (all prior D client work must be stable). Phase 0 prerequisite: rollup bugs fixed before log shipping tests are written.
