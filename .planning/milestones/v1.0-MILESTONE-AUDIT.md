## Unsatisfied Requirements — Detail

### PRE-3: executeBoundedDependency — not sole definition

**Phase:** 00-prerequisites
**Plan:** 00-03-PLAN.md
**Blocker:** Two extra definitions exist after extraction:

1. `modules/integrations/redis-client.js:9` — local wrapper named `executeBoundedDependency` that delegates to the shared module's import at line 4. The shared module is imported (`bounded-dependency.js`), but a local wrapper of the same name is then defined on top of it, so callers see the local wrapper, not the shared module directly.

2. `modules/routing/http-handler.js:86-141` — a fully inlined copy with its own constants (`DEPENDENCY_ATTEMPT_TIMEOUT_MS = 900`, `DEPENDENCY_TOTAL_TIMEOUT_MS = 1800`, `DEPENDENCY_RETRY_JITTER_MS = 120`). This copy does NOT import from `bounded-dependency.js` at all. It is passed as `options.executeBoundedDependency` into `createRedisClient` at `http-handler.js:139-141`, making this the actual implementation used in the production serverless request path for Redis.

**Phase goal:** `modules/integrations/bounded-dependency.js` is the sole definition; no duplicate definitions remain.
**Actual:** Three definitions exist.

---

### FR-5: Nightly failure log shipping — push path never wired

**Phase:** 04-nightly-log-shipping
**Plans:** 04-01-PLAN.md, 04-02-PLAN.md
**Blocker:** The nightly rollup route never calls `shipFailureLogs`. The phase delivered GET/DELETE `/operator/logs/pending` pull endpoints — useful but not the FR-5 deliverable.

**What's missing:**
- Inject `dClient` (or a `shipFailureLogs` callback) into the nightly rollup route dependency graph
- After `runNightlyRollup(...)` resolves, call `LRANGE quarantine:events 0 -1` and pass events to `dClient.shipFailureLogs(events)`
- Wrap the shipping call in try/catch; emit `logger.warn(...)` on failure; do not change response status
- Add sequencing, failure-isolation, and LRANGE non-destructiveness tests

**Both trigger sites are unwired:**
- `modules/routing/operator-routes.js:257-261` — `/operator/rollup/nightly` returns rollup result directly
- `modules/routing/request-controls.js:91-106` — `runNightlyMaintenance` calls rollup only
